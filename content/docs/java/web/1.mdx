> Cookie：浏览器提供的键值对存储载体，可自动随请求发送。
> Session：传统有状态认证方案，数据在服务端，客户端 Cookie 仅存 Session ID。
> JWT：现代无状态认证方案，数据自包含在令牌中，服务端无需存储会话。
## 对比表

| 维度           | Session（配合 Cookie）           | JWT（通常放 Cookie 或 Header） | Cookie（本身）                   |
|:---------------|:---------------------------------|:-------------------------------|:---------------------------------|
| **本质**       | 服务端会话管理机制               | 自包含的令牌格式（JSON+签名）  | 客户端存储/传输机制              |
| **数据存放**   | 服务端（内存/Redis/DB）          | 客户端（令牌内包含用户信息）   | 客户端（浏览器）                 |
| **状态性**     | 有状态（服务端维护会话）         | 无状态（服务端不存会话）       | 无状态（纯数据载体）             |
| **网络开销**   | 仅传输 Session ID（极小）        | 每次携带完整 JWT（较大）       | 每次请求自动携带所存数据         |
| **分布式支持** | 需共享存储（如 Redis）或黏性会话 | 天然支持，无需同步状态         | 无特殊要求                       |
| **主动注销**   | 服务端删除 Session，立即生效     | 难吊销（需黑名单/短期有效期）  | 清除 Cookie 即可                 |
| **典型用途**   | 传统 Web 应用、后台管理系统      | SPA、移动端、微服务、跨域 API  | 携带凭证、存偏好设置、防追踪限制 |


## 关键差异详解

### 1. 存储与状态管理
- **Session**：用户登录后，服务端生成 Session 数据并分配 `session_id`，通过 `Set-Cookie` 下发。后续请求浏览器自动携带该 ID，服务端查库/缓存获取用户状态。
- **JWT**：登录成功后，服务端将用户信息、过期时间等打包成 JSON，经签名生成 JWT 返回客户端。后续请求客户端携带 JWT，服务端**仅验证签名**即可信任内容，不查数据库。
- **Cookie**：只是“信封”。可以装 `session_id`，也可以直接装 JWT，或存语言偏好等无关认证的小数据。

### 2. 安全性对比
| 攻击类型     | Session + Cookie                        | JWT（存 localStorage）         | JWT（存 HttpOnly Cookie）     |
|:-------------|:----------------------------------------|:-------------------------------|:------------------------------|
| **XSS**      | ✅ `HttpOnly` 可防 JS 读取 ID            | ❌ JS 可读取完整令牌            | ✅ `HttpOnly` 防读取           |
| **CSRF**     | ❌ 自动携带，需 CSRF Token 或 `SameSite` | ❌ 若放 Cookie 同样有 CSRF 风险 | ❌ 同样需 `SameSite`/CSRF 防护 |
| **令牌泄露** | 仅泄露 ID，服务端可立即失效             | 泄露后可被长期冒用（难吊销）   | 同左，但可通过短有效期缓解    |

### 3. 扩展性与架构影响
- **Session**：在单机时代简单可靠，但在微服务/多节点部署时，必须引入 Redis 等共享存储或配置负载均衡黏性会话（Sticky Session），增加运维成本。
- **JWT**：完全无状态，任何节点均可验证，非常适合分布式架构、第三方授权、IoT 设备。但令牌体积通常比 Session ID 大 3~10 倍，高频请求可能增加带宽。

### 4. 吊销与生命周期管理
- **Session**：服务端主动删除即可，权限变更（如改密码、封号）实时生效。
- **JWT**：一旦签发，在过期前始终有效。要实现“强制下线”，常见做法：
  - 设置较短的 `exp`（如 15 分钟）
  - 配合 `Refresh Token` 机制
  - 服务端维护短期黑名单/版本号（牺牲部分无状态优势）
### 5. 选型建议
| 场景                                   | 推荐方案                        |
|:---------------------------------------|:--------------------------------|
| 传统多页网站、强权限管控、需频繁踢人   | Session + Redis                 |
| 前后端分离 SPA、移动端 App、微服务网关 | JWT + HttpOnly Cookie（防 XSS） |
| 跨域第三方授权、IoT/Serverless         | JWT（Authorization Header）     |
| 仅存用户偏好（主题/语言）              | 普通 Cookie                     |
